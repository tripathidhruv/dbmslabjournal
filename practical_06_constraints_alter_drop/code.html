Group A — Data Integrity

Practical 06: Constraints, ALTER, DROP

Managing database schemas post-creation and aggressively enforcing domain integrity guidelines.

Aim

Objective

The aim of this practical is to implement absolute data integrity using SQL constraints and to manage database schemas dynamically post-creation utilizing `ALTER` and `DROP` commands. The student will aggressively govern the Hospital constraints layer—enforcing Domain, Entity, and Referential locks—and analyze exact violation errors generated by the engine when malicious input is submitted.

Theory & Background

Core Concepts

The Role of Constraints in RDBMS

Constraints act as mathematically unbreachable firewalls situated immediately above the hard drive. They ensure data reliability by strictly enforcing conditions before any `INSERT` or `UPDATE` is physically written. If rules are violated, MySQL halts execution and throws an Exception. Entity Integrity ensures no two rows are identical (enforced via `PRIMARY KEY` or `UNIQUE`), while Domain Integrity ensures an isolated column value remains logically sound for that data type (enforced via `NOT NULL`, `CHECK`, and variable type limits).

Referential Integrity and Foreign Keys

When normalizing schemas, parent and child tables rely on relationship strings. Referential Integrity ensures that a child table cannot reference a parent id that does not genuinely exist in the parent table. Furthermore, if a DBA attempts to DELETE a parent row (e.g. Department), a strictly configured `FOREIGN KEY` will violently block the deletion (via `RESTRICT`) unless explicitly commanded to `CASCADE` the deletion down through all children, preventing 'orphaned data' from floating indefinitely in the DB.

ALTER TABLE: Dynamic Architecture Evolution

In a production matrix, destroying a table to add a new column is financially disastrous because it vaporizes the historical data within. `ALTER TABLE` is used to restructure active tables live. `ADD COLUMN` seamlessly pushes new attributes; `MODIFY COLUMN` shrinks or expands character limits (provided it doesn't truncate existing data); `RENAME COLUMN` updates architectural phrasing, and `DROP COLUMN` forcibly purges an entire column and its payload forever without locking the system.

Data Eradication: DELETE vs TRUNCATE vs DROP

`DELETE` is a DML (Data Manipulation) command; it destroys singular rows (leveraging `WHERE`), operates sequentially which is incredibly slow for massive data, and writes to log files meaning you can `ROLLBACK` to recover it. `TRUNCATE` is a DDL command; it physically tears down the internal table payload and rebuilds its hull instaneously, stripping all auto_increment counters down to 1. `DROP` destroys the hull, the framework, the payload, and completely frees up the hard disk memory—it completely vanishes the table entirely.

Procedure / Algorithm

Step-by-Step
  1. 01.Authenticate into the MySQL terminal and change environment to `USE hospital_db;`.
  2. 02.Attempt to forcefully insert a corrupted patient record that actively violates the established `CHECK` constraints.
  3. 03.Observe the strict failure trace returned by the MySQL daemon preventing bad integer ingestion.
  4. 04.Utilize `ALTER TABLE` to append a new column accommodating emergency contact data.
  5. 05.Morph the recently appended column utilizing `MODIFY` to expand character capacity limit.
  6. 06.Apply a `UNIQUE` lock on the new column preventing duplicates using `ADD CONSTRAINT`.
  7. 07.Insert a test Department record specifically built to test referential locking structures.
  8. 08.Attempt a `DELETE` against the test Department and observe how existing Foreign Keys react.
  9. 09.Eradicate the `UNIQUE` index constraint applying `DROP INDEX`.
  10. 10.Revert architectural modifications utilizing `DROP COLUMN`.
  11. 11.Initialize a dummy log table, and demonstrate instantaneous clearance applying `TRUNCATE TABLE`.
  12. 12.Conclude architecture manipulation by purging the temporary hull with `DROP TABLE`.

Complete MySQL Code

Runnable & Commented
MySQL 8.x — Hospital Management System 
-- Practical 06: Deep Dive Configuration of Constraints, ALTER, and DROP
USE hospital_db;

-- 1. CONSTRAINT VIOLATION TESTING
-- Let's try to pass an illegal negative age, triggering the internal generic CHECK.
/* 
   INSERT INTO PATIENT (patient_id, name, age) VALUES ('P999', 'Corrupt Data', -12);
   EXPECTED MYSQL ENGINE ERROR: Error Code 3819.
   Check constraint 'patient_chk_1' is violated. Insert is Halted entirely.
*/

-- 2. ARCHITECTURAL EXPANSION via ALTER
-- Adding a completely new attribute while preserving existing patient row data.
ALTER TABLE DOCTOR ADD COLUMN emergency_phone VARCHAR(15);

-- 3. MODIFYING EXISTING ARCHITECTURES
-- Realizing 15 characters is too short for international codes, we expand it to 20.
ALTER TABLE DOCTOR MODIFY COLUMN emergency_phone VARCHAR(20);

-- 4. ADDING LATE-STAGE CONSTRAINTS
-- Ensuring no two doctors register the exact identical emergency ping router.
ALTER TABLE DOCTOR ADD CONSTRAINT uq_doctor_emg UNIQUE (emergency_phone);

-- 5. RENAMING SCHEMA STRUCTURES
ALTER TABLE DOCTOR RENAME COLUMN emergency_phone TO contact_router;

-- 6. REFERENTIAL LOCKING ANALYSIS (FOREIGN KEYS)
-- Attempting to DROP the parent Cardiology Department (dept_id = 1).
/* 
   DELETE FROM DEPARTMENT WHERE dept_id = 1;
   EXPECTED ERROR: Error Code 1451. Cannot delete or update a parent row:
   a foreign key constraint fails (`hospital_db`.`DOCTOR`, CONSTRAINT ... )
*/

-- 7. CONSTRAINT ROLLBCAK / REDUCTION
-- Standard constraint drops target explicitly named indexes.
ALTER TABLE DOCTOR DROP INDEX uq_doctor_emg;

-- 8. ARCHITECTURAL CONTRACTION via DROP COLUMN
ALTER TABLE DOCTOR DROP COLUMN contact_router;

-- 9. DESTRUCTION TESTING: TRUNCATE VS DROP
-- Creating an arbitrary logging cache matrix.
CREATE TABLE ARBITRARY_LOG (
    id INT PRIMARY KEY AUTO_INCREMENT,
    ping VARCHAR(5)
);
INSERT INTO ARBITRARY_LOG (ping) VALUES ('A'),('B'),('C');

-- 9A. TRUNCATE leaves the hull structure intact but vaporizes data instantaneously.
TRUNCATE TABLE ARBITRARY_LOG;
-- 9B. We can prove the structure exists but Auto_Increment reset to 1.
INSERT INTO ARBITRARY_LOG (ping) VALUES ('X');
SELECT * FROM ARBITRARY_LOG;

-- 9C. DROP absolutely obliterates the table physically from disk.
DROP TABLE ARBITRARY_LOG;

Conclusion

Summary

Through precise interaction with SQL's definition layer, we demonstrated structural agility via ALTER statements, avoiding disastrous table reconstruction processes. We proved conclusively that standard `CHECK` and `FOREIGN KEY` constraints serve as impenetrable application guards against malicious or orphaned payload requests. Finally, by mapping out the lethal differences between TRUNCATE and DROP operations, we highlighted critical operational commands mandatory for system integrity upkeep.
