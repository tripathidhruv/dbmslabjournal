Group B — Data Retrieval

Practical 07: SQL Queries

Executing relational, boolean, subquery matrices, set structures, and aggregate query combinations extracting highly tuned datasets.

Aim

Objective

The aim of this practical is to master the execution of complex query parameters utilizing advanced structural Data Manipulation Language (DML). The student will compose heavily nested queries demonstrating `WHERE` comparisons, aggregate mathematics, strict conditional grouping (`HAVING`), Subqueries distributed across the execution stack, and combinatorial Set logical arrays.

Theory & Background

Core Concepts

Filter Operations & Query Architecture

The architecture of a `SELECT` limits exactly which data pipes stream from memory to interface. The `WHERE` clause fundamentally filters payload retrieval on a row-by-row level using operators: Relational (=, !=, >, <=) and Logical (AND, OR, NOT). Pattern strings are located using `LIKE` matrices, lists are scoped utilizing `IN`, ranges leverage `BETWEEN`, and nullity is detected strictly using `IS NULL`. Sequence ordering leverages `ORDER BY` and truncation leverages `LIMIT`.

Aggregate Analysis and Conditional Grouping

Raw data frequently lacks analytical clarity until grouped. Aggregate commands (`COUNT`, `SUM`, `AVG`, `MAX`, `MIN`) condense large sets mathematically. Critical rule: `WHERE` clauses cannot perform computation using Aggregates due to SQL execution order targeting `WHERE` *before* aggregation calculation. To filter aggregated grouped arrays, the `HAVING` clause explicitly operates sequentially after `GROUP BY` completes.

Subqueries and Nested Extraction

Subqueries are SQL statements embedded implicitly within the clauses of a root SQL statement. A nested subquery is executed only once and its calculated scalar output is injected instantly into the root statement's `WHERE` block. Conversely, a Correlated Subquery creates dramatic algorithmic drag as its embedded query mathematically references variables mapped directly to the root query, forcing the subquery engine to loop and execute individually *for every single row* traversing the outer logic execution matrix.

Set Combinatorial Operators

SQL enables directly stitching autonomous query payloads together completely bypassing relational ties via Set Operators. `UNION` glues distinct rows vertically into one super-column stack (purging exact overlap duplicates entirely). `UNION ALL` stacks queries insanely fast while preserving duplicates. While MySQL formally rejects strict `INTERSECT` and `MINUS/EXCEPT` syntaxes, seasoned DBA engineers simulate intersection matrices heavily relying on `IN` bindings nested through subqueries.

Complete MySQL Code

Runnable & Commented
MySQL 8.x — Hospital Management System 
-- Practical 07: Massive Query Expansions (Filtering, Subqueries, Sets, and Grouping)
USE hospital_db;

-- ────────── BLOCK 1: BOOLEAN FILTERS AND RELATIONAL SEQUENCES ──────────
-- A: Multi-conditional filtering matching wildcards using 'LIKE'
SELECT patient_id, name, disease FROM PATIENT 
WHERE (disease = 'Diabetes' OR disease = 'Migraine') 
AND name LIKE '%a%'
ORDER BY name ASC LIMIT 5;

-- B: Locating exactly nested range arrays leveraging BETWEEN and IN
SELECT doctor_id, name, salary, dept_id FROM DOCTOR 
WHERE salary BETWEEN 80000 AND 150000 
AND dept_id IN (2,3,4);

-- C: Distilling arrays parsing Nullity via IS NOT NULL
SELECT DISTINCT experience_years FROM DOCTOR 
WHERE experience_years IS NOT NULL ORDER BY experience_years DESC;

-- ────────── BLOCK 2: MATHEMATICAL AGGREGATION AND GROUPING ──────────
-- A: General aggregate functions computing financial blocks
SELECT COUNT(*) AS Total_Bills, 
       SUM(total_amount) AS Total_Cash_Generated, 
       MAX(total_amount) AS Heaviest_Bill 
FROM BILL;

-- B: Implementing STRICT GROUPS and post-aggregation HAVING constraints
SELECT department.dept_id, AVG(salary) AS Average_Departmental_Pay 
FROM DOCTOR 
GROUP BY dept_id 
HAVING Average_Departmental_Pay > 100000
ORDER BY Average_Departmental_Pay DESC;

-- C: Executing mathematical ROLLUP to print generic super-aggregates internally
SELECT gender, COUNT(*) AS Patient_Sum 
FROM PATIENT 
GROUP BY gender WITH ROLLUP;

-- ────────── BLOCK 3: SUBQUERY MANIPULATION EXTRACTS ──────────
-- A: Nested WHERE Subquery: Find Doctors making above the global institutional average
SELECT name, salary FROM DOCTOR 
WHERE salary > (SELECT AVG(salary) FROM DOCTOR);

-- B: Subquery located securely in FROM clause (Derived Internal Tables)
SELECT Max_Salary, Min_Salary 
FROM (SELECT MAX(salary) AS Max_Salary, MIN(salary) AS Min_Salary FROM DOCTOR) AS derived_pay_matrix;

-- C: Select subquery implementation for calculated output formatting
SELECT name, 
       (SELECT COUNT(*) FROM APPOINTMENT WHERE patient_id = PATIENT.patient_id) AS Appointments_Logged
FROM PATIENT LIMIT 4;

-- D: Subquery generating a Correlation matrix (Slow algorithmic execution)
-- Finding patients whose personal bill is strictly higher than their ward average.
SELECT patient_id, total_amount FROM BILL b1
WHERE total_amount > (
    SELECT AVG(total_amount) FROM BILL b2 
    WHERE b1.paid_status = b2.paid_status
);

-- ────────── BLOCK 4: SET OPERATORS ──────────
-- A: UNION combinations (Requires perfectly identical column structural definitions)
SELECT name AS Global_Entity_Directory, 'STAFF', doctor_id FROM DOCTOR
UNION
SELECT name, 'CIVILIAN', patient_id FROM PATIENT
ORDER BY Global_Entity_Directory ASC LIMIT 5;

-- B: Simulating INTERSECT (MySQL strictly lacks native INTERSECT structural support)
-- Identifying patients who have generated BOTH arbitrary appointments AND heavy bills.
SELECT patient_id FROM PATIENT 
WHERE patient_id IN (SELECT patient_id FROM APPOINTMENT)
AND patient_id IN (SELECT patient_id FROM BILL);
