Group A — Data Modelling

Practical 04: ER Diagrams

Designing and mapping an Entity-Relationship diagram for the Hospital Management System to a relational schema.

Aim

Objective

The aim of this practical is to design Entity-Relationship (ER) diagrams for the Hospital Management System — a structured graphical representation of all entities, attributes, and the relationships between them. The student will apply ER modelling notation (Chen's notation and Crow's Foot notation) to model real-world hospital data requirements before translating the conceptual model into a logical relational schema.

ER diagrams are the blueprint of any database system, serving as a communication tool between database designers, application developers, and business stakeholders. Understanding cardinality (one-to-one, one-to-many, many-to-many), participation constraints (total vs partial), and specialization/generalisation hierarchies is fundamental to database design and is tested extensively in examinations and interviews.

Theory & Background

Core Concepts

What is an ER Diagram?

An Entity-Relationship (ER) diagram is a visual representation of the logical structure of a database, proposed by Peter Chen in 1976. It models the real world as Entities (objects with independent existence), Attributes (properties of entities), and Relationships (associations between entities).

Entity Types

Strong Entity: has its own primary key (PATIENT, DOCTOR, DEPARTMENT). Weak Entity: cannot be uniquely identified without a parent entity — identified by a partial key + parent PK (e.g., APPOINTMENT may depend on PATIENT). Weak entities are shown with double rectangles in Chen notation.

Attribute Types

Simple: atomic, indivisible (age, salary). Composite: divisible into sub-parts (full_name → first_name, last_name). Multi-valued: can hold multiple values (phone_numbers) — shown with double ellipses. Derived: computed from other attributes (age derived from birth_date). Key Attribute: uniquely identifies entity — underlined in ER diagrams.

Cardinality Ratios

One-to-One (1:1): One doctor is head of one department. One-to-Many (1:N): One doctor has many appointments. Many-to-Many (M:N): Many patients can see many doctors — requires a junction/associative table (APPOINTMENT). Cardinality is one of the most important design decisions in ER modelling.

Participation Constraints

Total Participation (double line): every entity must participate in the relationship (every APPOINTMENT must have a PATIENT). Partial Participation (single line): some entities may not participate (some DOCTORs may have no appointments yet). Also called mandatory vs optional participation.

HMS ER Description

Entities: PATIENT, DOCTOR, DEPARTMENT, APPOINTMENT, BILL. Key Relationships: PATIENT makes APPOINTMENT (1:N) — one patient can have multiple appointments. DOCTOR handles APPOINTMENT (1:N). DEPARTMENT has DOCTOR (1:N). PATIENT generates BILL (1:N). DEPARTMENT is headed by DOCTOR (1:1 optional).

Mapping ER to Relational Schema

Rules: (1) Each strong entity → table with PK. (2) Each multi-valued attribute → separate table. (3) 1:N relationship → FK in the N-side table. (4) M:N relationship → new junction table with FKs from both entities. (5) Weak entity → table with composite PK (partial key + parent PK).

EER — Enhanced ER

EER extends ER with OOP concepts: Generalisation (PATIENT generalises to INPATIENT, OUTPATIENT — bottom-up), Specialisation (top-down), Aggregation (relationship treated as entity), and Category/union types. Disjoint vs Overlapping specialisation determines if a patient can be both in- and out-patient.

Procedure / Algorithm

Step-by-Step
  1. 01.Identify all entities in the Hospital Management System from the requirements: PATIENT, DOCTOR, DEPARTMENT, APPOINTMENT, BILL.
  2. 02.For each entity, list all attributes and classify them (simple, composite, multi-valued, derived, key).
  3. 03.Identify all relationships between entities and assign meaningful verb phrases: PATIENT makes APPOINTMENT, DOCTOR handles APPOINTMENT.
  4. 04.Determine cardinality for each relationship: PATIENT → APPOINTMENT (1:N), DOCTOR → APPOINTMENT (1:N), DEPARTMENT → DOCTOR (1:N), PATIENT → BILL (1:N), DEPARTMENT → DOCTOR as head (1:1).
  5. 05.Determine participation constraints: total participation — every APPOINTMENT must have a PATIENT and DOCTOR; partial — a DOCTOR may have zero appointments initially.
  6. 06.Draw the ER diagram using Chen's notation: rectangles for entities, ellipses for attributes, diamonds for relationships, lines for connections with cardinality labels.
  7. 07.Alternatively, use Crow's Foot notation (used in MySQL Workbench): lines with fork/crow's foot symbols for '1' and 'many' ends.
  8. 08.Identify the APPOINTMENT entity as potentially an associative (bridge) entity because it resolves the M:N relationship between PATIENT and DOCTOR.
  9. 09.Map the ER diagram to a relational schema: identify each table, primary key, foreign keys, and constraints.
  10. 10.Validate the schema: ensure every relationship is represented by either a FK or a junction table.
  11. 11.Open MySQL Workbench → Model → New Model → Add Diagram → Build the visual ER diagram using the drag-and-drop interface.
  12. 12.Export the diagram as PNG and SQL forward-engineering script — validate against the hospital_db schema.

Complete MySQL Code

Runnable & Commented
MySQL 8.x — Hospital Management System 
-- ================================================================
-- Practical 04: ER Diagram → Relational Schema Mapping
-- Hospital Management System — Complete Schema
-- ================================================================

-- ER Diagram Description (Text Representation):
--
-- [DEPARTMENT]--------[DOCTOR]--------[APPOINTMENT]
--      |                      |                          |
--   floor_no              salary                    appt_date
--   dept_name          specialization               status
--      |                      |                          |
--      +---------------+              [PATIENT]---+
--                                              |
--                                         [BILL]<--generates
--
-- Cardinalities:
-- DEPARTMENT : DOCTOR     = 1 : N (one dept has many doctors)
-- DOCTOR : APPOINTMENT    = 1 : N (one doctor has many appts)
-- PATIENT : APPOINTMENT   = 1 : N (one patient has many appts)
-- PATIENT : BILL          = 1 : N (one patient has many bills)
-- DEPARTMENT : DOCTOR(head)= 1 : 1 (optional, partial)

DROP DATABASE IF EXISTS hospital_db;
CREATE DATABASE hospital_db CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
USE hospital_db;

-- Strong Entity 1: DEPARTMENT (created first — no FK dependencies)
CREATE TABLE DEPARTMENT (
    dept_id        INT         PRIMARY KEY COMMENT 'Department unique identifier',
    dept_name      VARCHAR(50)  NOT NULL UNIQUE COMMENT 'Department name — must be unique',
    floor_no       INT         NOT NULL CHECK(floor_no BETWEEN 1 AND 20),
    head_doctor_id INT         DEFAULT NULL COMMENT '1:1 partial relationship to DOCTOR'
) ENGINE=InnoDB COMMENT 'Represents hospital departments';

-- Strong Entity 2: DOCTOR (FK to DEPARTMENT)
CREATE TABLE DOCTOR (
    doctor_id        INT          PRIMARY KEY,
    name             VARCHAR(80)  NOT NULL,
    specialization   VARCHAR(50)  NOT NULL,
    experience_years INT          NOT NULL CHECK(experience_years >= 0),
    salary           DECIMAL(10,2) NOT NULL CHECK(salary > 0),
    dept_id          INT,
    FOREIGN KEY (dept_id) REFERENCES DEPARTMENT(dept_id) ON DELETE SET NULL
    -- 1:N: One DEPARTMENT has many DOCTORs
) ENGINE=InnoDB;

-- Strong Entity 3: PATIENT
CREATE TABLE PATIENT (
    patient_id     VARCHAR(10) PRIMARY KEY,
    name           VARCHAR(80) NOT NULL,
    age            INT         NOT NULL CHECK(age BETWEEN 0 AND 150),
    gender         ENUM('Male','Female','Other') NOT NULL,
    disease        VARCHAR(100) NOT NULL,
    ward_no        INT,
    admission_date DATE        DEFAULT (CURRENT_DATE)
) ENGINE=InnoDB;

-- Associative (Junction) Entity: APPOINTMENT (resolves M:N between PATIENT & DOCTOR)
CREATE TABLE APPOINTMENT (
    appt_id    INT          PRIMARY KEY AUTO_INCREMENT,
    patient_id VARCHAR(10) NOT NULL,
    doctor_id  INT          NOT NULL,
    appt_date  DATE         NOT NULL,
    status     ENUM('Scheduled','Completed','Cancelled') DEFAULT 'Scheduled',
    FOREIGN KEY (patient_id) REFERENCES PATIENT(patient_id) ON DELETE CASCADE,
    FOREIGN KEY (doctor_id)  REFERENCES DOCTOR(doctor_id)  ON DELETE CASCADE
) ENGINE=InnoDB;

-- Dependent Entity: BILL (total participation — every bill must have a patient)
CREATE TABLE BILL (
    bill_id      INT          PRIMARY KEY AUTO_INCREMENT,
    patient_id   VARCHAR(10) NOT NULL,
    total_amount DECIMAL(10,2) NOT NULL CHECK(total_amount >= 0),
    paid_status  ENUM('Paid','Unpaid','Partial') DEFAULT 'Unpaid',
    bill_date    DATE         NOT NULL DEFAULT (CURRENT_DATE),
    FOREIGN KEY (patient_id) REFERENCES PATIENT(patient_id) ON DELETE CASCADE
) ENGINE=InnoDB;

-- Now add the self-referencing head_doctor FK to DEPARTMENT
ALTER TABLE DEPARTMENT
    ADD CONSTRAINT fk_head_doctor
    FOREIGN KEY (head_doctor_id) REFERENCES DOCTOR(doctor_id) ON DELETE SET NULL;

-- Verify schema structure
SHOW TABLES;
DESCRIBE APPOINTMENT;

Conclusion

Summary

This practical successfully demonstrated the process of ER modelling for the Hospital Management System. We identified five strong entities (PATIENT, DOCTOR, DEPARTMENT, APPOINTMENT, BILL), correctly classified all relationships with their cardinalities and participation constraints, and translated the conceptual ER model into a logical relational schema with appropriate primary keys, foreign keys, and integrity constraints.

The APPOINTMENT table serves as the bridge entity resolving the implicit many-to-many relationship between PATIENT and DOCTOR. The circular dependency between DEPARTMENT and DOCTOR (via head_doctor_id) was resolved by creating DEPARTMENT first, DOCTOR second, and then adding the FK with ALTER TABLE. This ordering strategy is a critical skill in schema deployment scripts and is a best practice for avoiding circular FK creation errors.

Viva Voce

Q & A
Q1.What is the difference between a strong entity and a weak entity?
Ans: A strong entity has its own unique primary key and exists independently (e.g., PATIENT with patient_id). A weak entity cannot be uniquely identified by its own attributes alone — it depends on a strong (owner) entity and uses a composite key of its partial key + owner's PK (e.g., an APPOINTMENT_DETAIL might depend on APPOINTMENT).
Q2.How do you map a Many-to-Many relationship in a relational schema?
Ans: A M:N relationship cannot be directly represented with just foreign keys. It must be resolved by creating a junction/associative/bridge table that contains foreign keys referencing both participating entities. APPOINTMENT is our bridge table with FKs to both PATIENT and DOCTOR.
Q3.What is the difference between total and partial participation?
Ans: Total (mandatory) participation — double line in ER — means every entity instance must participate in at least one relationship instance (every APPOINTMENT must have a PATIENT). Partial (optional) participation — single line — means some instances may not participate (a DOCTOR may initially have no APPOINTMENT).
Q4.What is a derived attribute? Give a hospital example.
Ans: A derived attribute is computed from other stored attributes and typically not stored physically. In our HMS, a patient's 'age' could be a derived attribute computed from their 'date_of_birth' (since storing only DOB is more accurate). Similarly, 'years_since_admission' can be derived from admission_date.
Q5.What is the difference between Chen's notation and Crow's Foot notation?
Ans: Chen notation: uses rectangles (entities), ellipses (attributes), diamonds (relationships), and 1/N/M labels on lines — good for academic/conceptual modelling. Crow's Foot: uses line-end symbols (|, O, <) to show cardinality and participation — more compact, widely used in tools like MySQL Workbench, ERwin, Lucidchart.
