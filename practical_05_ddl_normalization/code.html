Group B — Schema Design

Practical 05: DDL Commands & Normalization

Implementing MySQL DDL commands and applying normalization theory (1NF through BCNF) on the Hospital Management System.

Aim

Objective

The aim of this practical is to implement Data Definition Language (DDL) commands in MySQL to create, modify, and manage the Hospital Management System schema, and simultaneously apply database normalization techniques (1NF, 2NF, 3NF, BCNF) to eliminate data redundancy and ensure data integrity. The student will learn how improper schema design leads to update, insertion, and deletion anomalies, and how normalization resolves these systematically.

DDL is the backbone of database schema management — every production database begins with carefully written CREATE TABLE statements with appropriate constraints. Normalization is the theoretical framework that guides these design decisions, and understanding it is essential for designing efficient hospital databases where patient records, billing, and medical data must be stored without duplication or inconsistency.

Theory & Background

Core Concepts

DDL Commands Overview

DDL (Data Definition Language) commands define and manage the structure of database objects. Key commands: CREATE (creates table/db/index/view), ALTER (modifies existing structure — add/drop/modify columns), DROP (permanently deletes object), TRUNCATE (removes all rows but keeps structure), RENAME (renames object). DDL is auto-committed in MySQL.

Data Types in MySQL

Numeric: INT, BIGINT, DECIMAL(p,s), FLOAT, DOUBLE. String: VARCHAR(n), CHAR(n), TEXT, ENUM, SET. Date/Time: DATE, TIME, DATETIME, TIMESTAMP, YEAR. Binary: BLOB, BINARY. JSON: native JSON type (MySQL 5.7+). Spatial: POINT, LINESTRING, POLYGON. Choosing the correct data type impacts storage and performance significantly.

Normalization — What & Why?

Normalization is the process of organising a relational database to reduce data redundancy and improve data integrity. Proposed by E.F. Codd (1970). An un-normalised table can suffer from: Update Anomaly (changing a doctor's salary requires updating multiple rows), Insert Anomaly (cannot add a doctor without a patient), Delete Anomaly (deleting last patient loses doctor information).

1NF — First Normal Form

Rules: (1) Each column must contain atomic (indivisible) values — no repeating groups or arrays. (2) Each row must be uniquely identifiable by a primary key. Violation example: storing multiple phone numbers in one column (phone: '9876543210, 9123456789'). Fix: create a separate PATIENT_PHONE table with patient_id FK.

2NF — Second Normal Form

Rules: Must be in 1NF + No partial dependencies (non-key attributes must depend on the ENTIRE composite primary key). Relevant for tables with composite PKs. Example: if APPOINTMENT had (patient_id, doctor_id) as composite PK and stored doctor_name, that violates 2NF because doctor_name depends only on doctor_id (partial dependency). Fix: separate DOCTOR table.

3NF — Third Normal Form

Rules: Must be in 2NF + No transitive dependencies (non-key attributes must not depend on other non-key attributes). Example: if DOCTOR table stored dept_name and dept_floor_no, then dept_floor_no → dept_name → doctor_id (transitive). Fix: separate DEPARTMENT table, store only dept_id FK in DOCTOR.

BCNF — Boyce-Codd Normal Form

BCNF is a stricter version of 3NF: for every non-trivial functional dependency X → Y, X must be a superkey. BCNF eliminates anomalies that 3NF may not catch when there are overlapping candidate keys. Most tables in the HMS schema are already in BCNF after applying 3NF rules.

4NF & 5NF (Higher Normal Forms)

4NF: No multi-valued dependencies (a doctor with multiple specializations AND multiple languages should have separate tables rather than all combinations). 5NF (PJNF): No join dependencies — relevant to very complex many-to-many relationships. In practice, most production schemas target 3NF/BCNF.

Procedure / Algorithm

Step-by-Step
  1. 01.Use the hospital_db database: USE hospital_db;
  2. 02.Create all 5 tables using CREATE TABLE with appropriate data types, PRIMARY KEY, NOT NULL, and CHECK constraints.
  3. 03.Use DESCRIBE to inspect the created table structure: DESCRIBE PATIENT;
  4. 04.Demonstrate normalization by starting with an un-normalised PATIENT_DOCTOR_UNNORMALISED table.
  5. 05.Identify all anomalies in the un-normalised table: update anomaly (changing doctor salary affects many rows), insert anomaly, delete anomaly.
  6. 06.Apply 1NF: ensure atomic values — move multi-valued attributes to separate tables.
  7. 07.Apply 2NF: remove partial dependencies — each non-key column must depend on the full PK.
  8. 08.Apply 3NF: remove transitive dependencies — dept_name and floor_no moved from DOCTOR to DEPARTMENT.
  9. 09.Verify the normalised schema results in the final 5-table design with FK constraints.
  10. 10.Use SHOW CREATE TABLE to view the complete DDL including all FK definitions: SHOW CREATE TABLE APPOINTMENT\G
  11. 11.Create indexes on frequently queried columns to optimise performance.
  12. 12.Document all functional dependencies identified during normalization in your journal.

Complete MySQL Code

Runnable & Commented
MySQL 8.x — Hospital Management System

-- Practical 05: DDL Commands & Normalization
-- Hospital Management System
-- ================================================================

USE hospital_db;

-- ── Step 1: Demonstrate Un-Normalised Table (Violates 1NF & 2NF) ──
-- Un-Normalised: patient info + doctor info + dept info in one table
-- Anomalies present:
--   Update: Changing Dr. Sharma's salary requires updating ALL rows
--   Insert: Cannot add a doctor without a patient
--   Delete: Removing P001 deletes Dr. Sharma's information
CREATE TABLE UNNORMALISED (
    patient_id    VARCHAR(10),
    patient_name  VARCHAR(80),
    patient_age   INT,
    diseases      VARCHAR(200),  -- VIOLATION: Multi-valued — 'Diabetes, Hypertension'
    doctor_id     INT,
    doctor_name   VARCHAR(80),
    salary        DECIMAL(10,2),
    dept_name     VARCHAR(50),   -- Transitive: doctor_id → dept_id → dept_name
    dept_floor    INT
);

-- ── Step 2: Apply 1NF ──
-- Remove multi-valued 'diseases' — each patient has one disease per row
-- Ensure atomic values in all columns
-- (Our final PATIENT table already satisfies 1NF)

-- ── Step 3: Apply 2NF ──
-- Separate DOCTOR information (no partial dependencies in FK tables)

-- ── Step 4: Apply 3NF — Create DDL for HMS ──
-- 3NF achieved: dept_name moved to DEPARTMENT table
-- DOCTOR stores only dept_id FK (not dept_name — removing transitive dep)

DROP TABLE IF EXISTS UNNORMALISED;

-- Final Normalised Schema — Complete DDL
CREATE TABLE DEPARTMENT (
    dept_id        INT         PRIMARY KEY,
    dept_name      VARCHAR(50)  NOT NULL UNIQUE,
    floor_no       INT         NOT NULL CHECK(floor_no BETWEEN 1 AND 20),
    head_doctor_id INT         DEFAULT NULL
) ENGINE=InnoDB;

CREATE TABLE DOCTOR (
    doctor_id        INT         PRIMARY KEY,
    name             VARCHAR(80) NOT NULL,
    specialization   VARCHAR(50) NOT NULL,
    experience_years INT         DEFAULT 0 CHECK(experience_years >= 0),
    salary           DECIMAL(10,2) CHECK(salary > 0),
    dept_id          INT,
    FOREIGN KEY (dept_id) REFERENCES DEPARTMENT(dept_id) ON DELETE SET NULL
) ENGINE=InnoDB;

CREATE TABLE PATIENT (
    patient_id     VARCHAR(10) PRIMARY KEY,
    name           VARCHAR(80) NOT NULL,
    age            INT         NOT NULL CHECK(age BETWEEN 0 AND 150),
    gender         ENUM('Male','Female','Other') NOT NULL,
    disease        VARCHAR(100) NOT NULL,
    ward_no        INT         CHECK(ward_no > 0),
    admission_date DATE        NOT NULL DEFAULT (CURRENT_DATE)
) ENGINE=InnoDB;

CREATE TABLE APPOINTMENT (
    appt_id    INT          AUTO_INCREMENT PRIMARY KEY,
    patient_id VARCHAR(10) NOT NULL,
    doctor_id  INT          NOT NULL,
    appt_date  DATE         NOT NULL,
    status     ENUM('Scheduled','Completed','Cancelled') DEFAULT 'Scheduled',
    FOREIGN KEY (patient_id) REFERENCES PATIENT(patient_id) ON DELETE CASCADE,
    FOREIGN KEY (doctor_id)  REFERENCES DOCTOR(doctor_id)  ON DELETE CASCADE
) ENGINE=InnoDB;

CREATE TABLE BILL (
    bill_id      INT          AUTO_INCREMENT PRIMARY KEY,
    patient_id   VARCHAR(10) NOT NULL,
    total_amount DECIMAL(10,2) NOT NULL CHECK(total_amount >= 0),
    paid_status  ENUM('Paid','Unpaid','Partial') DEFAULT 'Unpaid',
    bill_date    DATE         NOT NULL DEFAULT (CURRENT_DATE),
    FOREIGN KEY (patient_id) REFERENCES PATIENT(patient_id) ON DELETE CASCADE
) ENGINE=InnoDB;

-- Create indexes on frequently searched columns
CREATE INDEX idx_patient_disease  ON PATIENT(disease);
CREATE INDEX idx_appt_date         ON APPOINTMENT(appt_date);
CREATE INDEX idx_bill_paid_status   ON BILL(paid_status);

-- Verify all tables and structure
SHOW TABLES;
DESCRIBE PATIENT;

Conclusion

Summary

This practical established the complete normalised DDL schema for the Hospital Management System. By progressing from an un-normalised monolithic table through 1NF (atomic values), 2NF (no partial dependencies), and 3NF (no transitive dependencies), we arrived at the final 5-table design in BCNF. Each normalization step was justified by identifying specific data anomalies and eliminating the functional dependencies that caused them.

Key takeaway: Normalization improves data integrity and reduces redundancy but may increase the number of JOIN operations required for queries. In practice, database designers sometimes deliberately denormaize (controlled redundancy) for read-heavy systems like data warehouses or reporting databases, while keeping OLTP databases like our HMS fully normalised to ensure consistency. Performance indexes were created on disease, appt_date, and paid_status columns for optimal query performance.

Viva Voce

Q & A
Q1.What is a functional dependency? Give an example.
Ans: A functional dependency X → Y means the value of X uniquely determines the value of Y. Example: patient_id → name (knowing patient_id tells us exactly which patient name). doctor_id → salary (knowing doctor_id determines the salary). Functional dependencies are the foundation of normalization theory.
Q2.What is a transitive dependency and how does 3NF eliminate it?
Ans: A transitive dependency occurs when A → B → C (A determines B which determines C, making A indirectly determine C). In our schema, if DOCTOR stored dept_name: doctor_id → dept_id → dept_name. 3NF removes this by moving dept_name to the DEPARTMENT table, making DOCTOR store only dept_id FK.
Q3.What is the difference between DROP TABLE and TRUNCATE TABLE?
Ans: DROP TABLE permanently deletes the table structure along with all data and releases storage (DDL, auto-committed). TRUNCATE TABLE removes all rows but keeps the table structure intact, resets AUTO_INCREMENT, and is also DDL (fast, no per-row logging). DELETE FROM removes rows one by one with individual transaction log entries (DML, can be rolled back).
Q4.Why is BCNF considered stricter than 3NF?
Ans: 3NF allows functional dependencies where the determinant is not a superkey, as long as the dependent attribute is part of some candidate key. BCNF eliminates even these: for every non-trivial FD X → Y, X must be a superkey. This handles anomalies in tables with overlapping candidate keys that 3NF cannot.
Q5.What does the CHECK constraint do? Does MySQL fully enforce it?
Ans: CHECK constraints in MySQL 8.0.16+ are fully enforced and will reject rows where the condition evaluates to FALSE. Example: CHECK(age BETWEEN 0 AND 150) prevents inserting age=200. Before 8.0.16, MySQL parsed but ignored CHECK constraints — a dangerous behaviour no longer present in current versions.
