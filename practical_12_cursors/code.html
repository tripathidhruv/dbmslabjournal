Group C — Row-Level Iteration

Practical 12: Cursors

Applying row-by-row procedural processing constructs within stored functions.

Aim

Objective

The aim of this practical is to study and implement Cursor mechanisms inside Stored Procedures. The student will understand how to bypass SQL's default set-based nature to iterate over result sets row-by-row, permitting complex procedural operations on individual rows retrieved from the database.

Theory & Background

Core Concepts

What is a Cursor?

Normally, SQL operates on sets (all matching rows at once). A cursor allows you to iterate over a query result set row-by-row. Think of it as a pointer that loops through the results of a SELECT statement in procedural PL/SQL code.

Lifecycle of a Cursor

1. DECLARE (define the query), 2. OPEN (execute query and load results into memory), 3. FETCH (pull one row from memory into variables), 4. CLOSE (free memory).

Handlers (NOT FOUND)

MySQL cursors don't natively know their length bounds. You must define a CONTINUE HANDLER FOR NOT FOUND to gracefully break the loop when FETCH no longer finds any more records to pull.

Procedure / Algorithm

Step-by-Step
  1. 01.Change the delimiter to allow block structure.
  2. 02.Create a stored procedure (e.g., UpdateAllDoctorSalaries).
  3. 03.Declare variables to hold row data (e.g., v_doc_id, v_salary).
  4. 04.Declare a variable (done = 0) loop-breaker and declare the cursor for the SELECT query.
  5. 05.Declare the NOT FOUND handler to set done = 1.
  6. 06.Open cursor, begin a LOOP, Fetch into variables.
  7. 07.Add an IF done THEN LEAVE loop conditional.
  8. 08.Perform row-based action (e.g., giving 5% raise).
  9. 09.Close cursor and end process.

Complete MySQL Code

Runnable & Commented
MySQL 8.x — Hospital Management System 
-- Practical 12: Cursors
USE hospital_db;

DELIMITER //

CREATE PROCEDURE ApplyBonusToDoctors()
BEGIN
    -- Variables to hold fetched data
    DECLARE v_doc_id INT;
    DECLARE v_salary DECIMAL(10,2);
    
    -- Loop control variable
    DECLARE done INT DEFAULT 0;
    
    -- Cursor Declaration
    DECLARE doctor_cursor CURSOR FOR 
        SELECT doctor_id, salary FROM DOCTOR;
        
    -- NOT FOUND handler: trips when cursor reaches end of dataset
    DECLARE CONTINUE HANDLER FOR NOT FOUND SET done = 1;
    
    -- Open the Cursor
    OPEN doctor_cursor;
    
    read_loop: LOOP
        -- Fetch current row into variables
        FETCH doctor_cursor INTO v_doc_id, v_salary;
        
        -- Check if we reached the end
        IF done = 1 THEN
            LEAVE read_loop;
        END IF;
        
        -- Procedural logic for each row: e.g., 5% bonus
        UPDATE DOCTOR
        SET salary = v_salary * 1.05
        WHERE doctor_id = v_doc_id;
        
    END LOOP;
    
    -- Close the connection and free memory
    CLOSE doctor_cursor;
    
END //
DELIMITER ;

-- Run the procedure containing the cursor
CALL ApplyBonusToDoctors();

-- Verify changes
SELECT doctor_id, name, salary FROM DOCTOR ORDER BY salary DESC;

Conclusion

Summary

We employed a database cursor successfully to bridge the gap between relational set logic and program loop logic. Handling individual rows via OPEN, FETCH, and CLOSE statements gave fine-grain procedural control otherwise difficult to execute via raw declarative SQL updates safely.

Viva Voce

Q & A
Q1.What is the primary difference between normal SQL queries and Cursors?
Ans: SQL acts on sets of data simultaneously. Cursors iterate sequentially over records row-by-row, permitting procedural control flow logic like loop conditional branching per record.
Q2.Why isn't a cursor recommended for large bulk updates?
Ans: Since cursors fetch row-by-row, they carry significant overhead and execute DML operations singly instead of in optimized batches. Normal UPDATE queries are almost always orders of magnitude faster.
Q3.What does the CONTINUE HANDLER do in a cursor?
Ans: It prevents the stored procedure from throwing a terminal error when a FETCH retrieves nothing. Instead, it flips a variable status (e.g. done = 1) allowing the code to break the loop naturally.
Q4.What are the mandatory sequential steps for a cursor?
Ans: DECLARE the cursor -> OPEN the cursor -> FETCH rows in a LOOP -> CLOSE the cursor.
Q5.Can you update data directly through a cursor?
Ans: Yes, databases often support updating the currently fetched record using the syntax UPDATE ... WHERE CURRENT OF cursor_name, though explicit updates matched by ID (like our example) are more universally compatible syntax.
