Group C: Advanced SQL Database Consistency

Triggers in
PL/SQL

Automating database integrity. Triggers allow us to execute procedural code automatically in response to specific events on a particular table or view.

bolt Core Concept

A trigger is a stored procedure that is automatically invoked by the DBMS when a specific event occurs. Unlike a subprogram, which must be invoked explicitly, a trigger is "fired" by the system.

BEFORE Triggers

Execute before the triggering statement runs. Often used to validate or modify data before it's saved.

AFTER Triggers

Execute after the statement completes. Ideal for logging, auditing, or cascading updates.

CREATE TRIGGER Syntax

-- Basic Structure
CREATE [OR REPLACE] TRIGGER trigger_name
{BEFORE | AFTER | INSTEAD OF}
{INSERT [OR] | UPDATE [OR] | DELETE}
[OF column_name]
ON table_name
[FOR EACH ROW]
[WHEN (condition)]
BEGIN
  -- Executable statements
END;

Row-Level Triggers

Specified with FOR EACH ROW. Fires once for each row affected by the SQL statement. Access old and new values using :OLD and :NEW.

Statement-Level Triggers

Default behavior. Fires exactly once for the SQL statement, regardless of how many rows are affected. Used for global integrity checks or logging the action itself rather than the data.

Practical Implementation: Audit Trail

-- Trigger to maintain Salary Audit
CREATE OR REPLACE TRIGGER audit_salary_update
AFTER UPDATE OF salary ON Employees
FOR EACH ROW
BEGIN
  INSERT INTO Salary_History
  (emp_id, old_sal, new_sal, change_date)
  VALUES(:OLD.id, :OLD.salary, :NEW.salary, SYSDATE);
END;

Mechanism Explanation

  • check_circle The trigger fires AFTER the record is updated.
  • check_circle It only fires when the salary column specifically is changed.
  • check_circle Using :OLD captures the state before the update for historical accuracy.
